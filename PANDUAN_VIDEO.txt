--- JEDA SEBENTAR ---

"Simulasi ini membangun ekosistem Cloud berskala kecil
 menggunakan Docker sebagai media provisioning
 sumber daya virtual. Ada tiga layer utama:"

  [Tunjukkan jari satu per satu]

  "Pertama - Layer IaaS atau Infrastructure as a Service.
   Direpresentasikan oleh container Docker yang berperan
   sebagai virtual machine atau compute instance."

  "Kedua - Layer PaaS atau Platform as a Service.
   Direpresentasikan oleh Redis sebagai managed cache service
   dan Prometheus sebagai managed monitoring service."

  "Ketiga - Layer SaaS atau Software as a Service.
   Yaitu aplikasi Flask yang bisa langsung diakses
   melalui browser tanpa perlu instalasi apapun."

--- PENJELASAN PRINSIP IDEAL ---

"Seluruh infrastruktur ini dibangun untuk membuktikan
 prinsip arsitektur cloud yang disebut IDEAL."

  "I - Isolated: setiap container Docker berjalan dalam
   namespace terpisah, mirip seperti virtual machine
   yang terisolasi secara penuh."

  "D - Democratic: Nginx mendistribusikan beban secara
   merata ke semua node menggunakan round-robin algorithm."

  "E - Elastic: kita bisa menambah node baru saat traffic
   meningkat, tanpa mematikan layanan yang sudah berjalan."

  "A - Adaptive: sistem otomatis mendeteksi node yang mati
   dan mengalihkan traffic ke node yang masih sehat."

  "L - Less-coupled: semua komponen berkomunikasi
   via HTTP dan Redis, tidak ada ketergantungan langsung."

"Sekarang mari kita buktikan teori ini dengan praktik!"

================================================================
FASE 1: RESOURCE PROVISIONING IaaS (2-3 menit)
================================================================

[Buka terminal, navigasi ke folder mini-cloud]

NARASI:
"Saya sekarang akan melakukan provisioning infrastruktur.
 Dalam cloud computing nyata, ini setara dengan
 mengklik tombol 'Launch Instance' di AWS atau GCP.
 Di sini kita gunakan Docker Compose."

KETIK DI TERMINAL:
  cd mini-cloud
  cat docker-compose.yml

SAMBIL KAMERA MENUNJUKKAN FILE:
"Di sini saya mendefinisikan seluruh infrastruktur
 sebagai kode - ini yang disebut Infrastructure as Code
 atau IaC. Docker Compose adalah versi sederhananya,
 sementara di industri digunakan Terraform atau CloudFormation."

"Perhatikan setiap service punya IP address statis,
 resource limits, dan health check - persis seperti
 konfigurasi cloud instance yang sesungguhnya."

KETIK DI TERMINAL:
  docker-compose up -d --build

SAMBIL BUILD BERJALAN:
"Proses ini mensimulasikan provisioning on-demand.
 Dalam hitungan detik, kita sudah memiliki:
 load balancer, dua worker node, cache server,
 dan monitoring system - semua berjalan paralel."

SETELAH SELESAI:
  docker-compose ps

"Lihat semua container statusnya 'Up' - infrastruktur
 sudah siap. Di cloud nyata ini artinya semua instance
 sudah running dan siap menerima traffic."

  docker stats --no-stream

"Kita bisa melihat penggunaan resource setiap node
 secara real-time. Ini seperti CloudWatch di AWS
 atau Cloud Monitoring di GCP."

================================================================
FASE 2: LOAD BALANCING & FAULT TOLERANCE (3-4 menit)
================================================================

NARASI:
"Sekarang kita buktikan prinsip Democratic - distribusi
 beban yang merata. Saya akan mengirim beberapa request
 dan lihat node mana yang melayani."

KETIK DI TERMINAL (satu per satu, beri jeda):
  curl http://localhost/ | python3 -m json.tool

"Perhatikan field 'node_id' - itu menunjukkan container
 mana yang melayani request ini."

  curl http://localhost/ | python3 -m json.tool

"Request kedua dilayani oleh node yang berbeda!
 Inilah round-robin load balancing - setiap node
 mendapat giliran yang sama dan adil."

--- DEMO CACHE HEADER ---
  curl -s -I http://localhost/static/asset

"Perhatikan header X-Cache-Status: MISS pada
 request pertama - artinya data diambil dari server."

  curl -s -I http://localhost/static/asset

"Request kedua: X-Cache-Status: HIT! Nginx
 sekarang melayani dari cache-nya sendiri.
 Ini simulasi CDN - Content Delivery Network."

--- DEMONSTRASI FAULT TOLERANCE ---

"Sekarang bagian paling penting: membuktikan High
 Availability dan Fault Tolerance. Saya akan
 mematikan satu node secara paksa."

  docker stop cloud-app-node-1

"Node 1 sudah mati. Dalam arsitektur tanpa HA,
 50% request akan gagal. Tapi dengan load balancer
 yang proper, mari kita lihat apa yang terjadi."

  curl http://localhost/ | python3 -m json.tool
  curl http://localhost/ | python3 -m json.tool
  curl http://localhost/ | python3 -m json.tool

"Semua request berhasil! Semuanya sekarang dilayani
 oleh app-node-2. Nginx mendeteksi node-1 tidak
 responsif dan secara otomatis mengalihkan semua
 traffic - inilah yang disebut automatic failover."

"Ini adalah inti dari prinsip Adaptive dalam IDEAL:
 sistem beradaptasi terhadap kegagalan tanpa
 intervensi manual dari administrator."

  docker start cloud-app-node-1

"Dan sekarang saya hidupkan kembali node-1.
 Dalam hitungan detik, Nginx akan mendeteksi
 node-1 sudah sehat dan memasukkannya kembali
 ke dalam pool - inilah self-healing."

  curl http://localhost/ | python3 -m json.tool
  curl http://localhost/ | python3 -m json.tool

"Kini distribusi kembali merata ke kedua node.
 Semua terjadi otomatis - zero downtime, zero
 human intervention."

================================================================
FASE 3: ELASTICITY & HORIZONTAL SCALING (2-3 menit)
================================================================

[SWITCH KE SCENE 1 sebentar - FACE CAM untuk penjelasan teori]

"Sebelum demo, izinkan saya menjelaskan perbedaan
 fundamental antara dua jenis scaling."

"Vertical Scaling atau Scale-Up berarti meningkatkan
 kapasitas server yang sudah ada - misalnya upgrade
 dari 4GB RAM ke 16GB RAM. Masalahnya ada downtime
 saat server direstart, dan ada batas hardware maksimal."

"Horizontal Scaling atau Scale-Out berarti menambah
 server baru ke dalam cluster. Tidak perlu downtime,
 tidak ada batas teoritis, dan biayanya lebih efisien
 karena bisa pakai commodity hardware."

"Cloud computing mempopulerkan horizontal scaling.
 AWS Auto Scaling, Kubernetes Horizontal Pod Autoscaler,
 semuanya berbasis konsep ini."

[SWITCH KE SCENE 2 - FACE + SCREEN]

KETIK DI TERMINAL:
  docker-compose ps

"Saat ini kita punya 2 worker node.
 Saya simulasikan lonjakan traffic dengan menambah
 node ketiga - ini disebut scale-out."

  docker-compose -f docker-compose.yml -f docker-compose.scale.yml up -d

"Perhatikan: service yang lain tidak terganggu sama
 sekali. Kita hanya menambah kapasitas, bukan
 merestart sistem."

  docker-compose -f docker-compose.yml -f docker-compose.scale.yml ps

"Sekarang ada 3 node aktif! Mari verifikasi
 distribusi traffic-nya."

  for i in 1 2 3 4 5 6; do curl -s http://localhost/ | grep node_id; done

"Lihat - request terdistribusi ke tiga node secara
 bergiliran. Kapasitas cluster naik 50% tanpa
 satu detik pun downtime."

FASE 4: CACHING & CDN SIMULATION (2-3 menit)
================================================================

NARASI:
"Fase terakhir: membuktikan peran Caching Layer
 dalam arsitektur cloud modern."

"Ada dua layer cache dalam simulasi ini:"

  "Layer pertama: Redis Application Cache.
   Menyimpan hasil komputasi berat agar tidak
   perlu diulang setiap request."

  "Layer kedua: Nginx Proxy Cache.
   Menyimpan response di edge node, seperti
   CDN yang men-cache konten di lokasi terdekat
   dengan pengguna."

  curl -s http://localhost/cache-demo | python3 -m json.tool

"Perhatikan: cache_hit: false dan message
 'data baru dicompute'. Ini CACHE MISS - server
 harus melakukan komputasi berat."

  curl -s http://localhost/cache-demo | python3 -m json.tool

"Request kedua: cache_hit: true! Data langsung
 dari Redis tanpa komputasi ulang. Response
 time turun drastis - dari ~500ms menjadi <5ms."

"Dalam konteks cloud: caching mengurangi beban server
 utama hingga 90%, menghemat biaya compute, dan
 meningkatkan throughput sistem secara signifikan."
